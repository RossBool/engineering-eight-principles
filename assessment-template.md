# 项目诊断报告模板

> 按八大原则逐项扫描，输出项目健康度评估和行动建议。
>
> **占位符说明**：所有 `{xxx}` 占位符需按以下指南填充。

---

# 📋 项目诊断报告

**项目名称**：`{project_name}` ← 项目正式名称，如"支付中台 v2.0"
**评估日期**：`{date}` ← YYYY-MM-DD 格式
**当前阶段**：Phase `{N}` — `{phase_name}` ← N=0-5，phase_name 取 phases.md 中的阶段名
**评估人**：`{assessor}` ← 执行评估的 agent 或人员名称

---

## 一、总体诊断

**一句话判断**：`{一句话概括}` ← 用"项目处于XX状态，最大风险是XX"句式，不超过30字

**总体评级**：🟢 健康 / 🟡 需关注 / 🔴 告警

| 维度 | 评级 | 关键发现 |
|------|------|--------|
| 架构边界 | 🟢/🟡/🔴 | `{finding}` ← 一句话，如"接口已文档化但2个模块存在循环依赖" |
| 技术杠杆 | 🟢/🟡/🔴 | `{finding}` ← 如"已识别缓存层为杠杆点，PoC 已验证" |
| 资源聚焦 | 🟢/🟡/🔴 | `{finding}` ← 如"同时在推3个方向，核心路径资源不足" |
| 系统韧性 | 🟢/🟡/🔴 | `{finding}` ← 如"支付链路有降级方案，消息队列无备份" |
| 质量底线 | 🟢/🟡/🔴 | `{finding}` ← 如"测试通过率96%，低于98%基线" |
| 交付节奏 | 🟢/🟡/🔴 | `{finding}` ← 如"最近3个sprint均按时交付" |
| 收尾能力 | 🟢/🟡/🔴 | `{finding}` ← 如"回滚方案已验证，但应急预案未演练" |
| 代价意识 | 🟢/🟡/🔴 | `{finding}` ← 如"2个被砍功能未通知相关方" |

## 二、八大原则逐项扫描

### 原则 1：三八线（架构边界）
**评级**：`{🟢/🟡/🔴}`
**检查项**：
- 服务/模块边界是否清晰？`{assessment}` ← "是/否/部分" + 补充说明
- 接口契约是否文档化？`{assessment}` ← "已文档化N个接口 / 未文档化"
- 是否存在跨边界的直接依赖？`{assessment}` ← "发现N处直接依赖：..."
**发现**：`{detailed_finding}` ← 2-3句详细描述，含具体例子
**建议**：`{recommendation}` ← 1-2条可执行的建议，含负责方

### 原则 2：侧翼登陆（杠杆点识别）
**评级**：`{🟢/🟡/🔴}`
**检查项**：
- 项目的关键杠杆点是什么？`{identified / not_identified}` ← 如已识别，写出杠杆点名称
- 杠杆点的技术可行性是否已验证？`{assessment}` ← "已PoC验证 / 未验证 / 部分验证"
- 是否有备选方案？`{assessment}` ← "有N个备选 / 无备选"
**发现**：`{detailed_finding}`
**建议**：`{recommendation}`

### 原则 3：长津湖（资源聚焦）
**评级**：`{🟢/🟡/🔴}`
**检查项**：
- 当前 sprint 的核心方向是否明确？`{assessment}` ← 写出核心方向名称或"未明确"
- 是否存在同时开辟多个核心模块的情况？`{assessment}` ← "是，正在同时推X和Y / 否"
- 关键路径上的资源是否集中？`{assessment}` ← "X人在关键路径 / 分散在N个方向"
**发现**：`{detailed_finding}`
**建议**：`{recommendation}`

### 原则 4：反绞杀（系统韧性）
**评级**：`{🟢/🟡/🔴}`
**检查项**：
- 核心链路是否有单点故障？`{assessment}` ← "有N个单点：... / 无单点"
- 熔断/降级/回滚机制是否就位？`{assessment}` ← "熔断已实现 / 降级已实现 / 回滚已实现"
- 监控告警覆盖率如何？`{assessment}` ← "覆盖N% / 覆盖率未知"
**发现**：`{detailed_finding}`
**建议**：`{recommendation}`

### 原则 5：上甘岭（质量底线）
**评级**：`{🟢/🟡/🔴}`
**检查项**：
- 质量基线是否定义？`{assessment}` ← "已定义N项基线 / 未定义"
- 当前各项质量指标是否达标？`{assessment}` ← "N项达标 / N项未达标：..."
- 是否有基线失守未恢复的情况？`{assessment}` ← "有N项跌破底线未修复 / 无"
**发现**：`{detailed_finding}`
**建议**：`{recommendation}`

### 原则 6：边打边谈（交付节奏）
**评级**：`{🟢/🟡/🔴}`
**检查项**：
- 迭代交付节奏是否稳定？`{assessment}` ← "每N周交付一次 / 节奏不稳定"
- 需求变更是否有管控流程？`{assessment}` ← "有变更评审流程 / 无 /  informal"
- 是否存在需求冻结/分级机制？`{assessment}` ← "已分级 / 未分级"
**发现**：`{detailed_finding}`
**建议**：`{recommendation}`

### 原则 7：金城（收尾能力）
**评级**：`{🟢/🟡/🔴}`
**检查项**：
- 上线方案是否完备？`{assessment}` ← "方案已编写 / 未编写 / 部分"
- 回滚和应急预案是否已演练？`{assessment}` ← "已演练N次 / 未演练"
- 复盘和知识沉淀机制是否存在？`{assessment}` ← "有复盘流程 / 无"
**发现**：`{detailed_finding}`
**建议**：`{recommendation}`

### 原则 8：代价意识
**评级**：`{🟢/🟡/🔴}`
**检查项**：
- 被砍/延期的功能是否已通知相关方？`{assessment}` ← "已通知 / 未通知 / 部分通知"
- 团队成员是否了解决策背后的取舍？`{assessment}` ← "已在团队会议中说明 / 未说明"
- 是否有被牺牲的技术探索需要后续补偿？`{assessment}` ← "有N项待补偿 / 无"
**发现**：`{detailed_finding}`
**建议**：`{recommendation}`
