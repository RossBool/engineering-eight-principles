# 工程八原则 (Engineering Eight Principles)

> 从大型复杂系统的规划与交付经验中提炼的软件工程方法论。
> 核心思想：**多线并行推进，边界清晰可控，以交付推动共识，以共识稳定交付。**

## Quick Start

首次使用时，问用户3个问题来路由模式：

1. **项目是新立项还是进行中？** → 新立项从 Phase 0 开始，进行中先做现状诊断
2. **你希望我做什么？**
   - 诊断现状 → **现状诊断**模式
   - 规划未来 → **阶段规划**模式
   - 全局视图 → **项目仪表盘**模式
   - 告诉我该做什么 → **阶段导航**模式
3. **项目目前最大的痛点是什么？** → 用于在八大原则中优先扫描相关维度

如果用户直接说"帮我用工程八原则看看这个项目"，默认走**现状诊断**模式。

## 方法论总纲

### 八大原则

| # | 代号 | 工程含义 | 工程映射 | 一句话 |
|---|------|---------|---------|--------|
| 1 | **三八线** | 服务间必须有清晰的接口契约和隔离边界 | 架构边界 & 接口契约 | 服务之间必须有"隔离带"——清晰的接口契约，各自独立演进 |
| 2 | **侧翼登陆** | 找到系统瓶颈的杠杆点，一个决策改变全局走向 | 杠杆点识别 & 技术破局 | 找到那个能让全局翻转的技术决策，集中资源打穿它 |
| 3 | **长津湖** | 资源有限时，集中力量攻克最关键的一个模块 | 极端约束下聚焦突破 | 选完切入点后，把优势资源集中在最关键的局部（先侧翼选点，再长津湖聚焦） |
| 4 | **反绞杀** | 核心链路必须多路冗余：熔断、降级、自动恢复 | 系统韧性 & 容错设计 | 核心链路必须有多路备份：熔断、降级、自动恢复 |
| 5 | **上甘岭** | 质量基线不可退让，SLA/性能/安全每项都要守住 | 质量底线寸步不让 | SLA、性能基线、安全标准——这些基线丢了就输了 |
| 6 | **边打边谈** | 保持迭代交付节奏，用已交付成果推动需求收敛 | 交付与需求同步演进 | 不要等需求完美再动手，迭代交付本身就是最好的需求协商 |
| 7 | **金城** | 上线前全链路压测和应急预案，临门一脚决定成败 | 临门一脚决定成败 | 上线前的全链路压测和应急预案，值得投入50%的精力 |
| 8 | **代价意识** | 每个技术决策都有隐性成本，被砍功能和被忽略的声音要可见 | 每个工程决策都有代价 | 被砍的功能、被边缘化的成员、被牺牲的技术探索——都要看见并承认 |

### 适用性自检

使用前先判断是否适用本方法论：

| 场景 | 适用？ | 建议 |
|------|--------|------|
| 3人以上团队，工期2个月+ | ✅ 完整使用 | 走六阶段全流程 |
| 2-3人小团队，工期1个月 | ⚡ 精简使用 | 只用三八线+上甘岭+边打边谈三原则，跳过阶段流程 |
| 单人项目或工期<2周 | ❌ 不适用 | 方法论成本超过项目本身 |
| 纯运维/Bug修复 | ❌ 不适用 | 无明确起止点 |
| 探索性原型（可能废弃） | ⚡ 部分使用 | 只用到 Phase 0 + Phase 1 |
| 大规模技术栈迁移 | ⚡ 配合使用 | 同时参考 long-march-engineering 技能 |

### 六阶段生命周期

```
立项研判 → 迭代冲刺 → 反绞杀 → 上甘岭 → 边打边谈 → 金城
(立项)    (迭代交付)  (韧性加固) (质量攻坚) (持续交付)  (上线收尾)
```

每个阶段有明确的**入口条件、核心动作、出口门禁**。详见 [phases.md](phases.md)。

## 使用入口

### 意图识别

用户来找你时，先识别意图再选择模式：

| 用户说 | 模式 | 输出产物 |
|--------|------|--------|
| "帮我看看这个项目怎么样" / "项目诊断" / "健康度评估" | **现状诊断** | 项目诊断报告 (Markdown) |
| "帮我规划这个项目" / "执行方案" / "怎么做" | **阶段规划** | 阶段执行方案 (Markdown) |
| "给我一个全局视图" / "项目仪表盘" / "状态图" | **项目仪表盘** | 项目状态图 (HTML) |
| "这个阶段该做什么" / "迭代节奏" / "下一步" | **阶段导航** | 阶段评估 + 行动建议 |

### 模式一：现状诊断（项目诊断报告）

**步骤：**
1. 收集项目信息（阶段、团队、技术栈、已知风险）
2. 按八大原则逐项扫描，给出 🔴🟡🟢 三色评级
3. 识别当前所处阶段，评估阶段入口/出口条件
4. 输出报告，模板见 [assessment-template.md](assessment-template.md)

### 模式二：阶段规划（执行方案）

**步骤：**
1. 确认项目背景和目标
2. 按六阶段拆解任务，每阶段定义里程碑
3. 识别"侧翼登陆点"（关键杠杆）和"上甘岭基线"（质量底线）
4. 输出方案，模板见 [campaign-template.md](campaign-template.md)

### 模式三：项目仪表盘（项目状态图）

**步骤：**
1. 收集项目各维度数据
2. 生成单文件 HTML 状态图，含雷达图+阶段进度+风险热力图
3. 模板见 [dashboard-template.html](dashboard-template.html)
4. 将收集到的数据注入 `window.__EIGHT_PRINCIPLES_DATA__`

### 模式四：阶段导航

**步骤：**
1. 确认当前阶段
2. 读取 [phases.md](phases.md) 中该阶段的详细指导
3. 检查入口条件是否满足、出口门禁是否可达
4. 输出当前阶段行动清单和下一阶段准备建议

## 关键决策框架

### 三八线划定法（架构边界）

当需要划定服务/模块边界时：
1. **识别相关方** — 列出所有利益相关方（团队、服务、外部系统）
2. **划定分界线** — 每个方的职责范围用一句话说清
3. **建立隔离契约** — 定义接口契约（API schema、事件协议、数据格式）
4. **各自独立演进** — 双方都不直接依赖对方内部实现

### 侧翼登陆识别法（杠杆点）

当项目陷入僵局时：
1. **常规方案成本几何？** — 当前方案的 ROI 是否可持续
2. **替代切入点在哪里？** — 有没有一个被忽视的突破口
3. **切入后果断推进** — 一旦选定，集中资源打穿，不要犹豫

### 边打边谈协议法（需求协商）

当需求不断变化时：
1. **交付不停** — 保持迭代交付节奏，每2周交付可验证的增量
2. **评审中有依据** — 用已交付的功能作为需求协商的依据
3. **需求分级管理** — 把需求变更分为：冻结需求(不可变)、待协商需求(需评审)、灵活需求(可调整)

## 反模式清单

| 反模式 | 工程教训 | 工程后果 |
|--------|---------|--------|
| 范围蔓延 | 项目边界被反复突破 | 需求无限扩张，核心目标失焦 |
| 资源堆砌陷阱 | 人力堆上去但问题没解决 | 资源堆砌不等于解决问题，需要方法 |
| 单点故障级联 | 关键链路无冗余 | CI/CD 单点故障导致全链路瘫痪 |
| 上线前需求突袭 | 即将交付时插入大量变更 | 上线前突然加需求，打乱全部节奏 |
| 复盘缺失 | 项目结束后不做总结 | 经验白白流失，下个项目重蹈覆辙 |
