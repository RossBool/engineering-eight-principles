# 六阶段操作手册

> 三八线协议 — 项目生命周期各阶段的详细操作指南

---

## Phase 0: 立项研判（启动决策）

**工程背景**：项目启动前，负责人需要评估是否值得投入资源，明确最小成功标准和最坏情况承受力。

### 入口条件
- 收到项目需求或业务机会

### 核心动作

**1. 启动决策三问**
- **不做项目的代价是什么？** — 项目不做会损失什么市场/用户/竞争力？
- **启动的底线目标是什么？** — 不是追求完美，项目的最小成功标准是什么？
- **最坏情况能否承受？** — 项目失败的最大损失是什么？

**2. 力量评估矩阵**

| 维度 | 评估项 | 达标标准 |
|------|--------|--------|
| 人力 | 团队人数 & 技能覆盖 | 关键岗位无缺口 |
| 基础设施 | 预算 & 基础设施 | 至少覆盖前3个sprint |
| 需求清晰度 | 需求清晰度 & 竞品了解 | 核心用户场景已验证 |
| 外部依赖 | 外部依赖 & 合作方 | 关键依赖已确认 |
| 时间窗口 | 市场时机 | 错过窗口的容忍度 |

**3. 输出**
- 项目可行性判断（Go / No-Go / Conditional Go）
- 底线目标声明（一句话）
- 初始风险评估

### 出口门禁
- [ ] 底线目标已明确
- [ ] 团队核心成员已到位
- [ ] 至少有一个"侧翼登陆点"（技术可行性已验证）

---

## Phase 1: 迭代冲刺（迭代交付）

**工程背景**：项目通过多次迭代交付，从MVP到稳定基线，逐步完成核心功能的开发与集成。

### 入口条件
- Phase 0 出口门禁已通过
- 技术方案已初步设计

### 迭代节奏模板

> **最少3次迭代，按需增减。"回撤重整"是唯一必做阶段。**
>
> 核心模式是：首次→突破→推进→**回撤**→稳定。
> 小项目可压缩为3次（首次→突破→回撤稳定），大项目可扩展到7-8次。

| 迭代类型 | 工程目标 | 目标 | 节奏 | 关键动作 | 必做？ |
|----------|---------|------|------|---------|--------|
| **首次** | MVP 首次 | 建立MVP | 2-3周 | 核心链路跑通，可演示的原型 | ✅ |
| **突破** | 核心功能突破 | 达到核心基线 | 3-4周 | 主功能模块交付，集中突破 | ✅ |
| **推进** | 功能扩展 | 超越核心基线 | 3-4周 | 次要功能补齐 | 按需 |
| **回撤重整** | 技术债务清理 | 回退到稳定基线 | 2-3周 | 架构重整，性能优化 | ✅ **必做** |
| **稳定** | 集成稳定 | 链路稳定 | 3-4周 | 全功能集成测试，性能基线 | ✅ |

### 每次迭代的内部节奏

```
迭代前: 目标确认 → 资源集结 → 需求清晰度更新（1-2天）
迭代中: 主方向全力推进 → 辅助方向协同 → 每日站会（进度同步）
迭代后: 成果清点 → 问题评估 → 下次迭代准备（1-2天）
```

### 关键规则
- **每次迭代只聚焦一个主要方向** — 不要同时开辟多个主方向
- **"回撤重整"必须做** — 连续迭代后必须回撤重整，避免技术债务累积
- **迭代之间有"休整期"** — 至少2天，用于复盘和准备
- **先侧翼选点，再长津湖聚焦** — 先确定核心方向（侧翼登陆），再集中资源（长津湖）

### 出口门禁
- [ ] 所有迭代目标均已达成或明确标注延期
- [ ] 核心功能通过集成测试
- [ ] 性能基线已建立
- [ ] 技术债务清单已梳理并排期

## Phase 2: 反绞杀（韧性加固）

**工程背景**：系统核心功能稳定后，需要对关键链路进行韧性加固，确保在故障冲击下仍能保持可用。

### 入口条件
- Phase 1 出口门禁已通过
- 系统核心功能已稳定

### 核心动作

**1. 关键链路识别（关键链路梳理）**
- 列出所有核心用户链路
- 标注每条链路的单点故障
- 评估每条链路的"故障脆弱度"（单点被切断后的影响范围）

**2. 反绞杀三策**

| 策略 | 工程背景 | 工程实现 |
|------|---------|--------|
| **多路保障** | 多通道并行保障可用性 | 多活部署、服务降级、备用通道 |
| **快速自愈** | 故障发生后自动快速恢复 | 自动故障转移、自愈机制、快速回滚 |
| **渐进暴露** | 逐步暴露变更以降低风险 | 灰度发布、特性开关、渐进式上线 |

**3. 故障注入演练（混沌工程）**
- 对核心链路做故障注入测试
- 验证熔断/降级/回滚机制
- 记录每次演练的恢复时间（RTO）和数据损失（RPO）

### 出口门禁
- [ ] 核心链路无单点故障
- [ ] 故障注入测试全部通过
- [ ] RTO < 目标值，RPO < 目标值
- [ ] 监控告警覆盖率 > 90%

---

## Phase 3: 上甘岭（质量攻坚）

**工程背景**：系统经过韧性加固后，需要对质量基线进行严格守护，确保每项指标都经受住高并发压力测试的考验。

### 入口条件
- Phase 2 出口门禁已通过
- 系统韧性已验证

### 核心动作

**1. 质量基线清单（质量基线定义）**

| 基线项 | 指标 | 底线 | 跌破后果 |
|--------|------|------|--------|
| 功能正确性 | 测试通过率 | ≥ 98% | 用户核心流程中断 |
| 性能基线 | P99 响应时间 | < 目标值 | 用户体验劣化 |
| 安全底线 | 高危漏洞数 | = 0 | 数据泄露/合规风险 |
| 可用性 | SLA 达标率 | ≥ 目标值 | 服务不可用 |
| 数据一致性 | 数据校验通过率 | = 100% | 脏数据污染下游 |

**2. 上甘岭守则**
- **每个基线项都有负责人** — 质量基线的每一项都有明确的 owner
- **每天检查指标** — 每日质量指标看板更新
- **跌破基线的指标必须恢复** — 任何指标跌破底线，24小时内启动修复
- **不接受"先上线再优化"** — 质量基线失守，整个项目就输了

**3. 压力测试（模拟高并发压力）**
- 全链路压测（目标：3倍峰值流量）
- 安全渗透测试
- 异常场景覆盖测试

### 出口门禁
- [ ] 所有质量基线指标达标
- [ ] 全链路压测通过
- [ ] 安全审计完成且无高危漏洞
- [ ] 质量基线文档化并由全团队确认

---

## Phase 4: 边打边谈（持续交付与需求协商）

**工程背景**：核心质量达标后，项目进入持续交付与需求协商并行阶段，通过迭代交付推进需求落地，同时与相关方协商最终版本范围。

### 入口条件
- Phase 3 出口门禁已通过
- 核心质量已达标

### 核心动作

**1. 需求分类**

| 议题类型 | 需求分类 | 处理方式 |
|---------|---------|--------|
| **冻结需求** | 不可改变的核心边界 | 需求冻结，不接受变更 |
| **待协商需求** | 需要协商的重大分歧 | 需求评审会讨论，PM决策 |
| **灵活需求** | 可灵活调整的执行细节 | 开发团队自行决定 |

**2. 交付节奏**
- 保持2周一个迭代交付
- 每次交付都是协商依据（"我们已经做了X，Y的需求是否还需要？"）
- 需求变更必须走"协商流程"，不能由单方决定

**3. 协商僵局的处理**
- **僵局超过2个sprint** → 升级到"高层会议"（VP/CTO 决策）
- **交付过程中出现意外** → 立即召开"紧急对齐会议"
- **需求方临时毁约** → 用"金城"阶段回应（见Phase 5）

### 出口门禁
- [ ] 所有"冻结需求"已冻结
- [ ] 所有"待协商需求"争议已解决
- [ ] 最终版本已确认，进入发布准备

---

## Phase 5: 金城（上线收尾）

**工程背景**：项目进入最终上线收尾阶段，通过预发布验证、正式发布和全量推进三步完成交付，签署交付协议并移交运维。

### 入口条件
- Phase 4 出口门禁已通过
- 发布版本已确认

### 核心动作

**1. 金城执行三步**

| 步骤 | 工程动作 | 时间 |
|------|---------|------|
| **预发布验证** | 全链路压测 + 灰度10%流量 | D-7 ~ D-3 |
| **正式发布** | 正式发布 + 实时监控 | D-Day |
| **全量推进** | 全量放开 + 72小时护航 | D+1 ~ D+3 |

**2. 交付协议检查清单**
- [ ] 回滚方案已验证（72小时内可完全回退）
- [ ] 应急预案已演练（数据库、缓存、消息队列）
- [ ] 值班表已排定（7×24 前72小时）
- [ ] 监控大盘已就位（核心指标实时可见）
- [ ] 通知所有"相关方"（上下游团队、运营、客服）

**3. 签署交付协议（项目收尾）**
- **项目复盘**：做得好的、做得差的、意外收获
- **知识沉淀**（知识库归档）：文档、经验教训、技术方案归档
- **团队表彰**（突出贡献者认可）：突出贡献者表彰
- **运维移交**：系统进入运维态，交接运维团队

### 出口门禁
- [ ] 系统稳定运行72小时
- [ ] 项目复盘已完成
- [ ] 文档已归档
- [ ] 运维交接已完成

## 阶段跳转规则

1. **不可跳过阶段** — 每个阶段的出口门禁是下一阶段的入口条件
2. **可以合并阶段** — 小项目可将 Phase 2+3 合并、Phase 4+5 合并
3. **可以回退阶段** — 如果出口门禁未过但已进入下一阶段，必须回退
4. **循环允许** — Phase 4 和 Phase 1 之间可以循环（新需求 → 新一轮迭代）

### 循环进入规则（Phase 4 → Phase 1）

当产品进入下一版本迭代时，从 Phase 4 循环回 Phase 1：
- **入口条件**：Phase 4 出口门禁已通过 + 新版本需求已确认
- **继承项**：质量基线（不重置）、技术债务清单（允许新增）
- **重置项**：迭代目标（根据新需求重新定义）、资源分配（可调整）
- **出口门禁**：与首次 Phase 1 相同

### 小项目简化路径

| 团队规模 | 建议路径 | 可跳过 |
|---------|---------|--------|
| 5人+ / 3月+ | Phase 0→1→2→3→4→5 全流程 | 无 |
| 3-5人 / 1-3月 | Phase 0→1→(2+3合并)→(4+5合并) | 无，但合并阶段 |
| 2-3人 / 1月内 | Phase 0→1（只做首次+回撤稳定） | Phase 2-5 |
| 单人 / 2周内 | 只用三八线+上甘岭+边打边谈三原则 | 全部阶段流程 |
